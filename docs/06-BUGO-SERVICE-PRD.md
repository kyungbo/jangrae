# 처음장례 부고장 서비스 PRD

> 장례 정보 플랫폼 "처음장례"의 부고장 생성·공유·열람 서비스

---

## 1. 서비스 개요

### 1-1. 배경과 문제

장례 발생 시 상주/유족은 극도로 혼란한 상태에서 부고를 작성하고 전달해야 한다. 현재 대부분의 부고는:

- 카카오톡으로 텍스트+이미지를 수동 전송
- 같은 내용을 수십~수백 명에게 반복 발송
- 장례식장 위치, 조문 시간, 부의금 계좌를 매번 별도 안내
- 일정 변경 시 재전송이 불가능 (이미 보낸 이미지는 수정 불가)
- 조문객은 필요한 정보(약도, 주차, 화환 주문)를 다시 물어봐야 함

### 1-2. 해결 방향

**"3분 안에 부고장을 만들고, 링크 하나로 모든 정보를 전달한다."**

- 상주가 최소 정보만 입력하면 부고장 페이지가 생성됨
- 장례식장 DB(1,080곳)와 연동하여 주소·전화·지도 자동 입력
- 링크 기반이므로 일정 변경 시 실시간 반영 (재전송 불필요)
- 조문객에게 필요한 모든 정보를 한 페이지에 제공
- jangrae.com의 가이드 콘텐츠와 자연스럽게 연결

### 1-3. 핵심 지표 (Success Metrics)

| 지표 | 목표 (런칭 3개월) | 측정 방법 |
|------|------------------|----------|
| 부고장 생성 수 | 월 100건 | DB count |
| 생성 완료율 | 80%+ | 시작 → 완료 전환율 |
| 평균 생성 시간 | 3분 이내 | 타임스탬프 |
| 부고 페이지 조회 수 | 부고당 평균 50+ | GA4 |
| 카카오톡 공유 수 | 부고당 평균 10+ | 공유 이벤트 |

## 2. 사용자 정의

### 2-1. Primary: 부고 생성자 (상주/유족)

- **상황**: 가족의 사망 직후, 극도로 감정적이고 혼란한 상태
- **목표**: 알려야 할 사람들에게 빠르고 정확하게 부고를 전달
- **제약**: 시간 압박, 감정적 소진, IT 리터러시 다양
- **핵심 니즈**: 최소한의 입력으로 격식 있는 부고를 만들고 싶다

### 2-2. Secondary: 조문객 (부고 수신자)

- **상황**: 카카오톡/문자로 부고 링크를 받음
- **목표**: 장례식장 위치, 시간, 부의금 계좌 등 필요 정보를 한 번에 확인
- **제약**: 앱 설치 의사 없음, 빠르게 확인하고 싶음
- **핵심 니즈**: 한 페이지에서 모든 정보를 보고 바로 행동(길찾기, 화환 주문, 계좌 복사)

### 2-3. Tertiary: 장례 대행자 (장의사/상조 직원)

- **상황**: 유족 대신 부고를 만들어주는 역할
- **목표**: 신속하게 부고를 생성하고 유족에게 전달
- **제약**: 여러 건을 동시에 관리
- **핵심 니즈**: 반복 작업을 빠르게 처리할 수 있는 효율적 UI

## 3. 경쟁 분석

### 3-1. daqda.kr (추모닷컴) — 핵심 참고 서비스

**구조**: 고인 소개 → 장례 정보 → 부의금 계좌 → 위치/교통 → 추모 활동

**강점**:
- 포괄적 정보 제공 (위치, 교통, 조문 예절 가이드)
- 다채널 공유 (문자, 카카오톡, 링크 복사)
- 카카오맵/카카오내비/티맵 연동
- 근조화환 주문 연결
- 조문 순서 가이드 (방명록 → 부의금 → 분향 → 상주 인사)
- 실시간 채팅 지원 (Vaiv Chatbot)

**약점/차별화 기회**:
- 상조사 종속적 (독립적으로 부고 생성 불가)
- 장례식장 DB 연동 없음 (수동 입력)
- jangrae.com 같은 장례 정보 플랫폼과의 연계 없음
- 온라인 부의금 결제 미지원
- 부고 생성 과정의 UX가 공개되어 있지 않음 (B2B 모델로 추정)

### 3-2. 기타 경쟁 서비스

| 서비스 | 특징 | 한계 |
|--------|------|------|
| 부고닷컴 | 이미지 기반 부고 생성 | 링크 기반 아님, 수정 불가 |
| 셀프부고 | 간단한 부고 이미지 생성 | 단발성, 조문 정보 미통합 |
| 상조사 자체 부고 | 상조 가입자 전용 | 비가입자 사용 불가 |

### 3-3. 처음장례 차별점

1. **장례식장 DB 연동**: 1,080곳 자동완성 → 주소, 전화, 지도 자동 세팅
2. **장례 정보 플랫폼 연계**: 조문 예절, 사후 행정 등 콘텐츠로 자연 연결
3. **누구나 무료 생성**: 상조 가입 여부와 무관
4. **링크 기반 실시간 업데이트**: 일정 변경 시 재공유 불필요
5. **조문객 → jangrae.com 유입**: 부고 1건당 수십~수백 명의 신규 방문자

## 4. 기능 명세

### Phase 1: MVP (4주)

> "부고를 만들고, 공유하고, 열람한다"

#### 4-1. 부고 생성 (Admin/Creator)

**경로**: `jangrae.com/bugo/create`

**Step 1: 고인 정보**
```
- 고인 성함 (필수)
- 나이/생년 (선택)
- 성별 (선택)
- 고인 사진 (선택, 영정 사진 업로드)
```

**Step 2: 상주 정보**
```
- 상주 목록 (관계 + 이름, 최소 1명)
  예: 아들 홍길동, 딸 홍길순
- 대표 연락처 (선택, 비공개 가능)
```

**Step 3: 장례 정보**
```
- 장례식장 (자동완성 — 1,080곳 DB에서 검색)
  → 선택 시 주소, 전화, 네이버맵 자동 입력
  → DB에 없으면 직접 입력 가능
- 빈소 호실 (텍스트)
- 입관 일시 (날짜 + 시간 picker)
- 발인 일시 (날짜 + 시간 picker)
- 장지 (텍스트, 예: 서울시립승화원)
```

**Step 4: 부의금 계좌 (선택)**
```
- 계좌 목록 (은행 + 계좌번호 + 예금주, 최대 5개)
- "계좌 비공개" 토글 가능
```

**Step 5: 미리보기 & 발행**
```
- 부고장 미리보기 (모바일 뷰)
- 수정 가능
- "부고장 만들기" 버튼 → 고유 URL 생성
  → jangrae.com/bugo/{shortId}
- 공유 버튼: 카카오톡 / 문자 / 링크 복사
```

**인증**:
- 카카오 로그인 (필수 — 수정/삭제 권한)
- 로그인 시점: Step 5 발행 직전 (생성 과정은 로그인 없이 진행 가능)

#### 4-2. 부고 열람 (Viewer/조문객)

## 5. 기술 아키텍처

### 5-1. 현재 → 변경

```
현재 (정적 사이트)              변경 (하이브리드)
━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━━
Next.js SSG                   Next.js SSG + SSR/API Routes
로컬 TS 데이터                 로컬 TS + Supabase (부고 데이터)
인증 없음                      카카오 로그인 (부고 생성자만)
GA4만                         GA4 + 부고 이벤트 트래킹
```

### 5-2. 기술 스택 추가

| 영역 | 기술 | 이유 |
|------|------|------|
| DB | **Supabase** (PostgreSQL) | 무료 티어 충분, Auth 내장, Vercel 연동 |
| 인증 | **Supabase Auth + 카카오 OAuth** | 한국 사용자 90%+ 카카오 보유 |
| 이미지 저장 | **Supabase Storage** | 영정 사진 업로드 |
| OG 이미지 생성 | **@vercel/og** (Satori) | 카카오톡 공유 미리보기 자동 생성 |
| 지도 | **카카오맵 JS SDK** | 무료, 한국 주소 정확도 높음 |

### 5-3. 데이터 모델

```sql
-- 부고 (bugo)
create table bugo (
  id            uuid primary key default gen_random_uuid(),
  short_id      text unique not null,        -- URL용 짧은 ID (예: "a1b2c3")
  creator_id    uuid references auth.users,  -- 생성자 (카카오 로그인)
  
  -- 고인 정보
  deceased_name text not null,
  deceased_age  int,
  deceased_gender text,                      -- 'male' | 'female'
  deceased_photo_url text,                   -- 영정 사진
  
  -- 장례 정보
  hall_id       text,                        -- funeral_halls DB 참조
  hall_name     text not null,
  hall_address  text,
  hall_phone    text,
  hall_room     text,                        -- 빈소 호실
  hall_lat      float,
  hall_lng      float,
  
  encoffin_at   timestamptz,                 -- 입관 일시
  funeral_at    timestamptz not null,        -- 발인 일시
  burial_place  text,                        -- 장지
  
  -- 상태
  status        text default 'active',       -- active | archived | deleted
  view_count    int default 0,
  
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);

-- 상주 (mourner)
create table mourner (
  id        uuid primary key default gen_random_uuid(),
  bugo_id   uuid references bugo on delete cascade,
  relation  text not null,                   -- 아들, 딸, 부인 등

  is_main   boolean default false,           -- 대표 상주 여부
  phone     text,                            -- 선택, 비공개 가능
  sort_order int default 0
);

create table bugo_account (
  id           uuid primary key default gen_random_uuid(),
  bugo_id      uuid references bugo on delete cascade,
  bank_name    text not null,
  account_no   text not null,
  holder_name  text not null,
  sort_order   int default 0
);
```

## 6. UX 설계 원칙

### 6-1. 감정적 맥락을 고려한 설계

- **최소 입력**: 필수 필드는 고인 성함, 장례식장, 발인 일시 3개뿐
- **자동 완성**: 장례식장 선택만으로 주소/전화/지도 자동 세팅
- **단계별 진행**: 한 화면에 모든 필드를 보여주지 않음 (인지 부하 최소화)
- **저장 & 나중에 완성**: 중간 저장 후 나중에 이어서 완성 가능
- **실수 방지**: 발인 일시가 입관 일시보다 빠르면 경고

### 6-2. 톤 & 매너

- **컬러**: cream/navy 기본 톤 유지, 부고 페이지만 약간 어두운 톤
- **타이포**: 본문은 가독성 우선, 제목은 격식체
- **카피**: 존댓말, 과한 감정 표현 자제, 담백하고 정중한 톤
- **로딩/에러**: "처리 중입니다" 대신 구체적 안내 ("부고장을 만들고 있습니다")

### 6-3. 접근성

- 50~60대 사용자 고려: 충분한 글씨 크기 (최소 16px), 높은 대비, 큰 터치 타겟
- 다크모드: Phase 1에서는 미지원 (부고 특성상 밝은 배경이 격식에 맞음)
